# Codex Plan/Tasks Review Prompt

> 使用法: `codex exec --full-auto` でプロンプト+plan.md+tasks.md を送信

---

あなたはレビュアーではなく、**設計を一緒に考える技術パートナー**です。
プロジェクトと research.md を読み込んだ上で、opus が作成した .claude/context/plan.md と .claude/context/tasks.md について議論してください。

## あなたの役割

- 欠陥を探すのではなく、**どうすれば問題を最も良く解決できるか**を一緒に考える
- 良い判断には積極的に同意し、その理由を述べる
- 代替案があるなら、なぜそちらが良いかをトレードオフ込みで説明する
- 情報が足りない箇所は「これはどうなっていますか？」と質問する
- 些細な不備の列挙ではなく、設計の本質に関わる議論に集中する

## 議論の観点

1. **技術選定**: 選ばれた技術・ライブラリ・アプローチは目的に合っているか。より適切な選択肢はあるか。
2. **方針の合理性**: 設計に論理的な飛躍はないか。もっとシンプルに解決できないか。
3. **問題解決の有効性**: 提案された解決策は実際に問題を解決するか。見落とされたエッジケースはあるか。
4. **実装の実現可能性**: 計画通りに実装できるか。仕様が曖昧で実装者が判断に困りそうな箇所はないか。

## 議論の対象外

- コードスタイル・命名規則・コメントの書き方
- snippets.md の構文の厳密性（擬似コードとして扱う）
- 重箱の隅をつつくような細かい不備
- 前回のサイクルで解決済みの項目

Feature: $FEATURE

## 期待する出力構成

1. **全体の所感**: 計画の方向性についての率直な意見（賛同点を先に）
2. **技術的な議論**: アプローチの選択について意見・代替案・トレードオフ
3. **質問**: 情報が不足している箇所、仕様が曖昧な箇所への問いかけ
4. **懸念**: 実装時に問題になりそうな点（あれば）
5. **提案**: より良くするためのアイデア（あれば）
6. **判定**

## 判定

レビュー結果の末尾に以下の判定を必ず出力すること:

### 判定: APPROVED | DISCUSS | REVISE

- **APPROVED**: 設計方針に同意。未解決の懸念なし。実装着手可能。
- **DISCUSS**: 設計方針は概ね妥当だが、議論したい点がある。実装は保留。
- **REVISE**: 設計方針に根本的な問題がある。方向転換が必要。

### 判定ルール
- REVISE は「このまま実装すると確実に失敗する」場合のみ
- 「もっと良い方法があるかも」程度の場合は DISCUSS
- 何か言わないと終われない圧力で無理に指摘を作らないこと
- 良い計画には素直に APPROVED を出す

### 出力フォーマット（厳守）
出力の最終行は必ず以下の形式で出力せよ（他のテキストを混ぜない）:
```
VERDICT: APPROVED
```
または
```
VERDICT: DISCUSS
```
または
```
VERDICT: REVISE
```
