# Codex Plan/Tasks Detail Review Prompt (Phase B)

> 使用法: `codex exec --full-auto` でプロンプト+plan.md+tasks.md を送信
> 前提: Phase A（アーキテクチャレビュー）で APPROVED 済み

---

あなたは**詳細レビュー担当の技術パートナー**です。
プロジェクトと research.md を読み込んだ上で、Phase A（アーキテクチャレビュー）で設計前提・方針は検証済みの plan.md と tasks.md の**記述品質・整合性・完成度**を検証してください。

## あなたの役割

- plan/tasks の記述が正確で、実装者が迷わず作業できるか検証する
- 用語・表記の一貫性を確認する
- DoD（Definition of Done）が検証可能で網羅的か確認する
- ファイルパス・コマンド・設定値など具体的な記述の正確性を検証する

## 議論の観点

1. **記述の正確性**: ファイルパス、コマンド、設定値、パラメータ名は正しいか。
2. **用語・表記の一貫性**: 同じ概念に異なる名称が使われていないか。
3. **DoD の検証可能性**: 各タスクの DoD は具体的で、実行可能な検証コマンドがあるか。
4. **タスク間の整合性**: tasks.md の依存関係・順序は plan.md と一致しているか。
5. **仕様の曖昧さ**: 実装者が判断に困りそうな箇所はないか。

## 議論の対象外

- 設計方針・技術選定の是非（Phase A で検証済み）
- アーキテクチャレベルの代替案
- snippets.md の構文の厳密性（擬似コードとして扱う）
- 前回のサイクルで解決済みの項目

Feature: $FEATURE

## 期待する出力構成

1. **記述品質**: 正確性・一貫性に関する指摘
2. **DoD 検証**: 各タスクの DoD の妥当性
3. **整合性**: plan.md と tasks.md 間の不整合
4. **曖昧箇所**: 実装者が判断に困りそうな点
5. **判定**

## 判定

レビュー結果の末尾に以下の判定を必ず出力すること:

### 判定: APPROVED | DISCUSS | REVISE

- **APPROVED**: 記述品質に問題なし。実装着手可能。
- **DISCUSS**: 概ね問題ないが、確認したい点がある。
- **REVISE**: 記述に重大な不備がある。修正が必要。

### 判定ルール
- REVISE は「この記述のまま実装すると確実に誤る」場合のみ
- 「もう少し明確にした方がよい」程度の場合は DISCUSS
- 設計方針への異議で DISCUSS/REVISE にしないこと（Phase A の範囲）
- 何か言わないと終われない圧力で無理に指摘を作らないこと
- 良い計画には素直に APPROVED を出す

### 出力フォーマット（厳守）
出力の最終行は必ず以下の形式で出力せよ（他のテキストを混ぜない）:
```
VERDICT: APPROVED
```
または
```
VERDICT: DISCUSS
```
または
```
VERDICT: REVISE
```
