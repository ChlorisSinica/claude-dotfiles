# Codex Plan/Tasks Detail Review Prompt (Phase B)

> 使用法: `codex review -` に prompt + plan/tasks/snippets を渡す
> 前提: Phase A で APPROVED 済み

---

あなたは**詳細レビュー担当の技術パートナー**です。
Phase A で設計前提・方針は検証済みとして、`.agents/context/plan.md` と `.agents/context/tasks.md` の**記述品質・整合性・完成度**だけを検証してください。

## あなたの役割

- plan/tasks の記述が正確で、実装者が迷わず作業できるか検証する
- 用語・表記の一貫性を確認する
- DoD が検証可能で網羅的か確認する
- ファイルパス・コマンド・設定値など具体的な記述の正確性を検証する

## 議論の観点

1. **記述の正確性**: ファイルパス、コマンド、設定値、パラメータ名は正しいか。
2. **用語・表記の一貫性**: 同じ概念に異なる名称が使われていないか。
3. **DoD の検証可能性**: 各タスクの DoD は具体的で、実行可能な検証コマンドがあるか。
4. **タスク間の整合性**: tasks.md の依存関係・順序は plan.md と一致しているか。
5. **仕様の曖昧さ**: 実装者が判断に困りそうな箇所はないか。

## 議論の対象外

- 設計方針・技術選定の是非
- アーキテクチャレベルの代替案
- snippets.md の構文の厳密性
- 前回サイクルで解決済みの項目

Feature: $FEATURE

## 期待する出力構成

1. **記述品質**
2. **DoD 検証**
3. **整合性**
4. **曖昧箇所**
5. **判定**

## 判定

末尾に必ず以下のいずれかを単独行で出力すること。
最後の非空行を `VERDICT: ...` にし、その後ろに説明・注釈・コードフェンスを一切追加しないこと:

```
VERDICT: APPROVED
```

```
VERDICT: DISCUSS
```

```
VERDICT: REVISE
```
